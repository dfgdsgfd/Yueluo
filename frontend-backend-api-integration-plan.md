# 前端对接后端数据接口实施方案

## 1. 任务目标

为 `next-front-end` 前端项目全面对接 `backend-gin` 后端数据接口，完成真实接口替换、鉴权接入、数据结构适配、错误处理、上传、WebSocket、分页、缓存与性能优化等工作。

本次对接要求：

- 不遗漏已有前端页面、组件、业务流程中的数据接口。
- 不遗漏后端已实现且前端业务需要使用的接口。
- Swagger 作为接口发现和辅助参考。
- 实际后端实现代码为最终依据。
- 对接完成后前端应能正常启动、构建、联调和基本验收。
- 对无法对接或后端缺失的接口，必须形成明确说明。
- 兼顾性能优秀性，避免粗暴请求、重复请求、过度客户端渲染和大体积状态管理。
- 尽量避免改动前端既有布局、视觉结构和交互位置；如接口对接确实需要影响布局，必须标注原因、影响范围和验证结果。

## 2. 已知信息

- 后端目录：`backend-gin`
- 前端目录：`next-front-end`
- API 参考地址：`https://xse.yuelk.com/api/swagger-MYQD6LuH0heYgcK5DT10Al00dj6OW8Wc.json`
- API 参考用途：辅助梳理路由、鉴权、请求方法和模块归属。
- 最终依据：`backend-gin` 实际路由、handler、middleware、参数绑定和响应结构。

## 3. 核心原则

### 3.1 后端代码优先

接口文档可能存在字段不完整、示例缺失或与真实实现不同步的情况，因此执行顺序为：

1. 扫描 `backend-gin` 实际路由注册。
2. 阅读对应 handler / service / middleware。
3. 参考 Swagger 补全接口清单。
4. 与 `next-front-end` 页面和组件逐一匹配。

### 3.2 先矩阵，后实现

对接前先建立完整矩阵，避免边做边漏。

至少建立三张清单：

- 后端接口矩阵。
- 前端页面和功能矩阵。
- 前端现有请求、mock、硬编码数据矩阵。

### 3.3 统一请求层

所有 HTTP 请求必须通过统一请求层进入，避免页面中散落 `fetch` / `axios` 调用。

统一请求层负责：

- base URL。
- token 注入。
- 请求参数序列化。
- 响应结构拆包。
- 错误处理。
- 登录失效处理。
- 文件上传。
- 并发请求控制。
- 请求取消。
- 性能缓存策略入口。

### 3.4 性能优先

接口对接不是简单替换 URL。需要同时考虑：

- 请求数量是否过多。
- 是否重复请求。
- 首屏是否阻塞。
- 是否可以服务端渲染。
- 是否需要缓存。
- 列表是否需要分页、懒加载或虚拟滚动。
- 图片、视频、上传是否有合理优化。
- WebSocket 是否有重连和节流策略。

### 3.5 布局保护优先

接口对接原则上只改数据层、请求层、状态层和必要的 UI 状态，不主动重做页面布局。

允许改动的范围：

- 替换 mock 数据为接口数据。
- 增加 loading、empty、error 等必要状态。
- 修正因真实字段长度、图片比例、空值导致的展示问题。
- 增加分页、加载更多、上传进度等业务必需控件。
- 修复真实数据接入后暴露出的溢出、错位、重复渲染问题。

需要标注原因的布局影响：

- 调整页面结构。
- 调整 grid / flex 布局。
- 调整组件宽高。
- 调整间距、字号、颜色。
- 新增明显可见区域。
- 删除或移动已有区域。
- 改变按钮、表单、列表、卡片的展示位置。

每一处必须说明：

- 修改原因。
- 对应接口或真实数据问题。
- 影响页面。
- 影响范围。
- 是否有替代方案。
- 验证方式。

## 6. 阶段三：统一请求层设计

### 6.1 环境变量

建议统一使用：

```env
NEXT_PUBLIC_API_BASE_URL=https://xse.yuelk.com
```

如前端需要通过 Next.js 服务端代理，可根据项目结构增加：

```env
API_INTERNAL_BASE_URL=http://backend-gin:8080
```

最终命名以项目现有约定为准。

### 6.2 HTTP Client 能力

统一封装：

- `apiGet`
- `apiPost`
- `apiPut`
- `apiDelete`
- `apiUpload`
- `apiDownload`

统一处理：

- `Authorization: Bearer <token>`
- `Content-Type`
- query 参数
- path 参数
- `FormData`
- Blob 响应
- JSON 响应
- 业务错误码
- HTTP 错误码
- 登录过期跳转
- 请求超时
- 请求取消

### 6.3 响应结构适配

后端若统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

前端统一在请求层拆包：

- 成功：返回 `data`。
- 业务失败：抛出带 `code` 和 `message` 的业务异常。
- 未登录：清理登录态并跳转登录页。
- 权限不足：进入 403 处理或显示提示。

## 8. 阶段五：性能优秀性方案

### 8.1 请求性能

必须避免：

- 页面加载时重复请求同一接口。
- 父子组件各自请求同一数据。
- tab 切换反复请求未变化数据。
- 搜索框每输入一个字符立即请求。
- 无限滚动没有并发保护。

处理策略：

- 对稳定数据使用缓存。
- 对搜索输入使用 debounce。
- 对列表请求使用分页。
- 对 tab 数据使用按需加载。
- 对重复请求做请求合并或缓存复用。
- 对卸载组件中的请求做取消。
- 对 WebSocket 重连使用指数退避。

### 8.2 Next.js 渲染策略

根据页面性质选择：

| 页面类型 | 推荐策略 |
| --- | --- |
| 公共内容详情 | SSR 或服务端获取数据 |
| SEO 重要页面 | SSR / SSG / ISR，视后端数据变化频率决定 |
| 登录后个人数据 | 客户端请求或服务端鉴权后请求 |
| 管理后台 | 客户端请求为主，配合缓存 |
| 高频变化列表 | 客户端分页请求 |
| 低频配置数据 | 缓存或构建时加载 |

实际选择以项目现有架构为准。

### 8.3 数据缓存

如项目已有 SWR / React Query，优先沿用。

如没有现成方案，按最小侵入原则处理：

- 用户信息缓存。
- 分类标签缓存。
- App 配置缓存。
- 公告等低频数据缓存。
- 列表数据按 query key 缓存。
- mutation 后精准失效相关缓存。

缓存不能牺牲数据正确性，尤其是：

- 钱包余额。
- 提现状态。
- 审核状态。
- 未读消息数。
- 管理操作结果。

这些数据需要更短缓存或操作后立即刷新。

### 8.4 列表性能

列表页必须考虑：

- 分页。
- 加载更多。
- 骨架屏。
- 空状态。
- 错误状态。
- 筛选条件变化后的请求取消。
- 大列表虚拟滚动，若页面确实存在大量 DOM。
- 避免一次性请求全量数据。

### 8.5 图片与视频性能

处理策略：

- 使用 Next.js Image 或项目已有图片组件。
- 明确图片尺寸，避免布局抖动。
- 懒加载非首屏图片。
- 视频封面优先加载，视频本体按需加载。
- 头像、封面、缩略图使用合适尺寸。
- 避免前端直接加载原始超大图作为列表缩略图。

若后端提供缩略图字段，前端优先使用缩略图。

### 8.6 上传性能

处理策略：

- 上传前校验文件大小和格式。
- 显示上传进度。
- 防止重复点击提交。
- 上传失败允许重试。
- 大文件上传能力以后端实际支持为准。
- 若后端支持分片上传，再接入分片、断点续传。

### 8.7 Bundle 性能

检查并优化：

- 避免把后台管理大组件打入公共首屏。
- 富文本编辑器、图表库、视频播放器按需动态加载。
- 图标库按需引入。
- 大型工具函数避免整包导入。
- 移除对接后无用 mock 和测试数据。

### 8.8 状态管理性能

原则：

- 服务端数据不盲目塞进全局 store。
- 全局 store 只保存登录态、用户基础信息、全局配置等必要状态。
- 列表、详情、筛选条件优先放在页面或数据请求缓存中。
- mutation 后只刷新受影响数据。

## 9. 布局影响控制方案

### 9.1 默认不改布局

接口对接期间默认保持现有：

- 页面结构。
- 导航结构。
- 栅格和列表布局。
- 卡片结构。
- 表单排列。
- 按钮位置。
- 组件尺寸。
- 视觉风格。

如果只是数据字段名变化、请求方式变化、状态管理变化，应在数据层解决，不调整布局。

### 9.2 必须改布局的常见原因

只有以下情况才考虑改动布局：

- 后端真实数据字段比 mock 数据明显更长，导致文本溢出或遮挡。
- 后端真实图片、视频比例与前端假数据不一致，导致容器变形。
- 后端返回分页数据，原页面没有分页或加载更多入口。
- 上传接口需要进度、失败重试或文件状态展示。
- 权限或审核状态需要展示必要业务状态。
- 空数据、错误数据会导致页面塌陷或不可操作。
- 管理端批量操作、审核操作需要展示后端必需字段。

### 9.3 布局影响记录格式

如发生布局相关改动，最终交付时按以下格式记录：

| 页面/组件 | 改动内容 | 原因 | 对应接口 | 影响范围 | 验证结果 |
| --- | --- | --- | --- | --- | --- |
| 示例 | 增加分页控件 | 后端列表为分页返回，避免一次性加载全量数据 | `GET /api/...` | 列表底部 | 桌面和移动端无错位 |

## 11. 阶段七：鉴权与安全

需要确认：

- token 来源。
- token 存储方式。
- 是否有 refresh token。
- 管理员和普通用户是否共用登录体系。
- 是否需要区分 admin token 和 user token。
- 上传接口是否需要鉴权。
- WebSocket token 如何传递。
- API Key 接口是否前端可直接调用。

前端处理：

- 未登录跳转登录。
- 401 自动清理登录态。
- 403 展示无权限。
- 管理页增加权限保护。
- 敏感接口不在客户端暴露不该暴露的密钥。

- tab 切换是否复用缓存。
- 上传是否阻塞 UI。
- WebSocket 是否频繁重连。

### 12.4 布局验证

如果发生任何布局相关改动，必须验证：

- 相关页面桌面端无明显错位。
- 相关页面移动端无明显错位。
- 文本不溢出关键容器。
- 图片和视频比例正常。
- loading、empty、error 状态不破坏页面结构。
- 新增控件不遮挡原有操作。
- 改动原因已记录。

## 13. 交付物

最终交付：

- 完成接口对接的 `next-front-end` 代码。
- 接口对接矩阵。
- 前端页面对接矩阵。
- mock / 硬编码数据清理记录。
- 性能优化说明。
- 布局影响说明；若没有布局影响，明确标注“无布局影响”。
- 无法对接项说明。
- 环境变量说明。
- 启动与构建命令。
- 联调验证结果。

## 14. 执行顺序建议

1. 放入 `backend-gin` 和 `next-front-end`。
2. 扫描后端实际路由和 Swagger，生成接口清单。
3. 扫描前端页面、组件、hooks、store、mock。
4. 建立对接矩阵。
5. 设计或改造统一请求层。
6. 先接认证和用户信息。
7. 再接核心内容和互动模块。
8. 接上传、通知、WebSocket。
9. 接钱包、创作者中心和后台管理。
10. 清理全部 mock 和硬编码数据。
11. 做类型、字段和错误处理收口。
12. 做性能检查和优化。
13. 运行构建、lint、typecheck 和关键流程验收。
14. 输出最终对接报告。

## 15. 风险与处理方式

| 风险 | 处理方式 |
| --- | --- |
| Swagger 与后端实现不一致 | 以后端源码为准，并记录差异 |
| 前端已有 mock 分散 | 全局搜索并建立清理矩阵 |
| 鉴权逻辑不统一 | 先统一请求层和登录态 |
| 后端返回字段不稳定 | 建立类型和 mapper 适配 |
| 列表数据量大 | 分页、缓存、虚拟滚动按需处理 |
| 图片视频资源过大 | 使用缩略图、懒加载、明确尺寸 |
| 真实数据导致布局溢出或错位 | 优先数据层适配；必须改布局时记录原因和验证结果 |
| 管理端接口多 | 按模块矩阵验收，避免遗漏 |
| WebSocket 不稳定 | 心跳、断线重连、指数退避 |
| 后端缺少前端需要的接口 | 记录缺口，给出替代方案或待后端补充清单 |

## 16. 完成标准

本任务完成时，应满足：

- 前端所有已有业务页面均接入真实后端数据。
- 后端主要业务接口均完成归属分析。
- 所有 mock、假数据、硬编码接口均已处理或明确记录。
- 请求层、鉴权、错误处理、上传、分页、缓存机制统一。
- 关键业务流程可联调通过。
- 构建流程可运行。
- 性能检查无明显重复请求、过大首屏数据、无控制大列表、无严重 bundle 膨胀。
- 未无故改动前端既有布局；如有布局影响，已逐项标注原因、影响范围和验证结果。
- 对未完成项有明确、可追踪、可复测的说明。
